# 经验教训与设计原则

## 📅 日期：2026-03-14

---

## 一、技术教训

### 1. 前端技术栈确认
**教训**：一开始就假设前端是 Vue.js，写了 Vue 代码，结果前端是 React + TypeScript。
**原则**：
- 写代码前先确认技术栈（`package.json`、文件后缀 `.tsx` vs `.vue`）
- 看现有代码的 import 语句判断框架（`import React` vs `import Vue`）
- 不要凭经验假设，每次项目都验证

### 2. API 字段一致性
**教训**：API 返回 `name`，前端用了 `record.resource`，导致删除功能无效。
**原则**：
- 写前端前先确认 API 返回的字段名（用 curl 实际测试）
- 前后端字段命名保持一致，或用接口文档对齐
- TypeScript 接口定义应与 API 返回值严格匹配

### 3. URL 编码与特殊字符
**教训**：资源名 `API:GET:/users` 中的 `/` 被 URL 路径解析，导致 404。
**原则**：
- 用户输入的内容（资源名等）用 ID 操作，不用名称
- 路径参数中的用户输入必须 `encodeURIComponent()`
- 或改用查询参数 `?id=xxx` 避免路径编码问题

### 4. 数据库 NULL 处理
**教训**：MySQL 字段为 NULL 时，Go 扫描到 string 变量失败。
**原则**：
- 可空字段用 `sql.NullString`/`sql.NullInt64` 代替原生类型
- 或在 schema 设计时用 `NOT NULL DEFAULT ''` 避免 NULL
- 前端也要处理 null 值（空数组 vs null）

### 5. 依赖版本兼容性
**教训**：go-sql-driver/mysql v1.9.3 需要 Go 1.21+，项目用 Go 1.20。
**原则**：
- 添加依赖前检查版本兼容性
- 降级到兼容版本（v1.8.1）
- 用 `go mod vendor` 同步 vendor 目录

### 6. 前端构建产物
**教训**：改了前端代码但没重新构建，用户看不到变化。
**原则**：
- 前端代码改了必须 `npm run build`
- Dashboard 重启才能加载新的构建产物
- 给用户刷新链接时提醒「硬刷新」（Cmd+Shift+R）

## 二、设计原则

### 7. 用 ID 操作记录
**教训**：用名称（可能含特殊字符）做操作参数导致 URL 编码问题。
**原则**：
- 所有增删改操作用 ID（`record.id`）
- 名称仅用于展示
- 查询参数优于路径参数（`?id=xxx` vs `/:name`）

### 8. 确认弹窗要明确
**教训**：用户把确认弹窗「确定删除...吗？」当成报错。
**原则**：
- 确认弹窗要有明确的「确定/取消」按钮
- 错误信息和确认提示要区分开
- 成功后要有明确的成功提示

### 9. 模块设计无默认值
**教训**：「默认模块」设计增加复杂度，用户不需要。
**原则**：
- 模块就是模块，没有特殊的默认模块
- 资源可以没有模块（null/nil）
- 不自动创建任何特殊模块

### 10. 模块编辑边界
**教训**：用户对模块的修改有明确的边界需求。
**原则**：
- 模块名称创建后不可修改（避免引用断裂）
- 模块描述可以修改
- 有资源绑定的模块不能删除（防止误操作）

### 11. 配置文件与代码分离
**教训**：MySQL 连接信息写死在代码里，不方便切换环境。
**原则**：
- 所有配置（MySQL、etcd 等）放在 `conf/` 目录下的配置文件
- 代码里只读取配置，不硬编码
- 敏感信息（密码）用环境变量或配置文件

### 12. 端到端测试
**教训**：改了一个功能，另一个功能坏了。
**原则**：
- 每次改动后跑端到端测试（e2e_test.py）
- 覆盖所有核心流程（CRUD + 发布 + 回滚）
- 前端也要用浏览器实际测试

## 三、沟通原则

### 13. 定期汇报
**教训**：用户要求「中间过程记得汇报」。
**原则**：
- 复杂任务分阶段汇报（开始、中途、完成）
- 遇到问题立即反馈
- 不等全部完成才汇报

### 14. 先理解再动手
**教训**：用户说「第2点没有任何变化」，因为我理解错了需求。
**原则**：
- 复述用户需求，确认理解正确再动手
- 不确定时问清楚，别自己猜
- 验收标准要对齐

## 四、文件结构教训

### 15. 任务目录管理
**教训**：文件散落在各处，不好找。
**原则**：
- 所有任务在 `tasks/` 目录创建子目录
- 设计文档放 openspec
- 代码按项目规范放

### 16. 记录操作位置
**教训**：不知道文件改在哪里了。
**原则**：
- 记录关键文件路径
- 日志里记录操作位置
- commit message 描述清楚改了什么
